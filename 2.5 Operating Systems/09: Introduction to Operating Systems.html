OS Tasks
  1. Manages hardware and system resources
  2. Shares out and accounts for resources
  3. Offers secure environment for applications
  4. Provides common device or I/O system

Hardware Abstraction
  • Allows us to hide hardware differences from operating system
  • Allows us to run same OS on different hardware devices


Style of Operating Systems
Embedded / Real-Time
Proven long term reality, strict timing guarantees. Has a small footprint with no unnecessary code. Has certification for safety.
Server
Emphasis on fairly sharing resources and reliability. Can be same as Desktop but with different parameters.
Desktop
Focuses on interactivity and GUI. Could be the same as Server but with different parameters.


Some Versions
PC-DOS and MS-DOS
Control Program for Microcomputers (CP/M). Designed with limits of 8085 / PC - no code protection as its in assembly. Provided module / driver support which was hardly used.
Windows 1.0 (1985)
Non-overlapping windows, each app runs full screen and can be switched using tab and enter.
Windows 2.11 (1987)
Overlapping Windows and PS/2 mouse support. Intel 286 or 386 with Extended Memory driver.
MS-DOS 5.0
Menu driven interface where command prompt was still available.
Windows 3.1 (1992)
Added network based file and printer sharing. Virtual memory and later internet support.
Windows NT (1993)
32 bit OS independent of MS-DOS, collaboration of OS/2 with IBM. Remained a separate code base until Windows XP. Server and Workstations determined during installation of OS

Backward Compatibility: Ability to rollback to older version of operating system and everything still works as expected.


Monolithic OS: Entire OS is working within a kernel.
  • Traditional approach, tight coupling of non-application code, at most two protection levels (system and user).
  • However, couldn’t stop drivers introducing code to the system. Led to system errors being developed (blue screen) to protect the code.
  • Efficient structure due to shared access to resources.
  • Difficult to maintain as code boundaries often blurred - many ways of doing one thing.
          


Security: Layered approach - Multics Rings
  • Good concept with a good solution in terms of security.
  • Often not used at it means system is less efficient. Most OS simply distinct between user and system.



Microkernel OS
  • Less code running elevated privileges - most code is subject to security constraints.
  • Means that you have a restricted way the you obtain data - thus resulting in worse performance
  • Extensible and have minimal system dependencies - APIs make it easy.

Monolithic vs Microkernel OS


Modular
  • Kernel modules run with elevated privileges - more safety with regards to what information is passed.
  • Can access and share resources across modules.
  • Solves problem with regards to monolithic and microkernel OS limitation.


Library OS / Uni-Kernel
  • Kernel configures security at lowest level, libraries can get additional security.
