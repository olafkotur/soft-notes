Statecharts: Graphical formalism developed by David Harel for modelling reacting systems, allow you to trace behaviour of system, given specific inputs.

State Transition Networks (STN) - Finite State Machines
  • Good for modelling things as sequence of events that trigger transitions between states.
  • Not good at modelling things that have any parallelism that can lead to an “explosion of states” e.g. Twitter
  • Extends finite state machine formalism to support description of transitions within state in terms of nested states.

Definitions
Guard / Action Specification (GAS)
  • GAS is used in many parts of statechart specifications. 
  • GAS action is executed if guard is satisfied where
  • Guard = event_spec[condition_spec]
Event Specification
  • Expression in form of parentheses ‘&’ and ‘I’ of events. 
  • Satisfied when given combination of events is occurred at that moment. 
  • Should be instantaneous, thus sensed when occurred.
Condition Specification
  • Expression in form of the ‘&’ and ‘I’ of conditions.
Action Specification
  1. Generating events: ev1[condA]/ev2
  2. Assignments: sum := mathscore + englishscore
  3. Comma separated actions: ev1[condA]/ev2, ev3, a:=3
Transition Texts
  • All transitions except default transitions should have guard.
  • Default transitions doesn’t have guard and action is optional.
     
Condition Connector
  • Exactly one outgoing flow should be triggered when condition connector is entered.
     
Syntactic Rules
  • State boxes shouldn’t overlap.
  • When state has XOR children, one child should be destination of default transition.
  • Transition should start and end at state except default transition.
  • Transition can’t go through state which is decomposed into parallel children 
  • Transition can’t go through &-line 
Operational Semantics
  • Statechart executed step by step.
  • Every change in step makes effects on next step.
  • Lifetime of event is a single step and all events are broadcasted.
