Software Design Process: Set of requirements is transformed into collection of data structures and algorithms that implement software requirements as one or more computer programs.


Architectural design
The subsystems making up the system and their relationships are identified and documented. Model of control is identified.
Interface design
For each sub-system, it’s interface with other subsystems is designed and documented.
Component design
Services (funct reqs) allocated to different components and interfaces of these components are designed.
Data structure design
Data structures used in system implementation are designed in detail.
Algorithm design
Algorithm used to provide service are designed in and specified.



Key objectives of software design
  1. Correctness: Software should clearly conform to its specification
  2. Robustness: Software is robust if it can handle unusual conditions.
  3. Flexibility: User requirements are not static.
  4. Reusability: Engineering and economic motivation for reusability.
  5. Efficiency: Should execute as fast as required and use least amount of memory.
  6. Reliability: Software is reliable if it is free from significant defects.


Software Architecture: Set of structures needed to reason about the system. Structure is simple of elements held together by relations, includes static, dynamic and allocation structures.




Steps in Architectural Design
System structuring
Structured into a number of principal sub-systems or components and the communication between sub-systems identified.
Control modelling
General model of control relationships between parts of system is established.
Modular decomposition
Each sub-system is decomposed into modules - architect must decide on their interconnections e.g. objects, components. 

Box and Line diagrams
  • Boxes represent a sub-system where arrows represent function cals, data or control signals. If boxes within a box then sub-system has been decomposed to components.
  • Don’t show nature of relation between system elements.
  • Don’t show element’s externally visible properties.

Conceptual and Logical architecture
  • Conceptual: High level overview of subsystems, components and their interactions. Break apart system and get idea of what services and components system provides. 
  • Logical: Defines details of interfaces and interactions of services and components that make up application.
