Case Study: Design of a football game
  • User starts the game
  • User selects two teams
  • User adds / removes players to / from a team
  • User picks a ground to play at
  • User begins the match
Objects
Real world objects
  1. Player
  2. Team
  3. Ball
  4. Playground
  5. Referee
Logical objects
  1. Game: Defines football game
  2. GameEngine: Simulate number of games at time.
  3. TeamStrategy: Decide team’s tactics while playing.



Design Problem Sketch
  1. When ball changes position, all players and referees must be notified immediately.
  2. When game is in progress, user can change their team’s tactics at any time (Attack —> Defend).
  3. Player in team should have additional responsibilities that can be assigned at runtime (Forward <—> Defender).
Problem 1
When subject (ball) changes, all its dependants (players) are notified and undated automatically
  • Observer Pattern: Define one-to-many dependancy between objects so that when one object changes state, all dependants are notified and updated automatically.
Problem 2
Let algorithm (TeamStrategy) vary independently from clients (Team) that use it.
  • Strategy Pattern: Define family of algorithms, encapsulate each one, make them interchangeable. Strategy lets algorithm vary independently from clients that use it.
Problem 3
Attach additional responsibilities to object (Player) dynamically, without using sub classing.
  • Decorator Pattern: Attach additional responsibilities to an object dynamically. Decorators provide flexible alternative to sub classing for extending functionality.
