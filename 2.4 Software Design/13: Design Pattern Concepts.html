Design Pattern
  • Devs encounter problems of similar type when developing all sorts of software.
  • Thus solutions are similar across various types of applications (reuse).
  • Can use common vocabulary for efficient communication between devs.

Structure
  1. Name of pattern
  2. Description of problem that can be fixed
  3. Example of use
  4. Description of solution - Textual, UML, source code
  5. Discussion of positives  / negatives

Behavioural: Observer Pattern: Also known as publish / subscribe. Observe the state of an object, when subject changes state —> observes are informed, used by MVC paradigm.




Creational: Singleton Pattern: Situations where you need only one object of a class.




Structural: Adapter Pattern: Euro socket needs an adapter for a UK plug to work.

Applying Design Pattern
  • Designing solution and you run into a problem.
  • Designing a solution and notice that something looks like a pattern.

Design Patterns
Behavioural
Creational
Structural
Chain of responsibility: delegates commands to chain of processing objects.
Abstract Factory: groups object factories that have common theme.
Adapter: allows classes with incompatible interfaces to work together by wrapping its own interface around that of an already existing class.
Command: creates objects which encapsulate actions and parameters.
Builder: constructs complex objects by separating construction and representation.
Bridge: decouples abstraction from its implementation so that two can vary independently.
Interpreter: implements specialised language.
Factory Method: creates objects without specifying exact class to create.
Composite: composes one-or-more similar objects so that they can be manipulated as one object.
Iterator: accesses elements of an object sequentially without exposing its underlying representation.
Prototype: creates objects by cloning existing project.
Decorator: dynamically adds / overrides behaviour in existing method of object.
Mediator: allows loose coupling between classes by being only class that has detailed knowledge of their methods.
Singleton: restricts object creation for class to only one instance.
Facade: provides small interface to large body of code.
Memento: provides ability to restore an object to its previous state.

Flyweight: reduces cost of creating and manipulating large number of similar objects.
Observer: publish / subscribe pattern which allows number of observer object to see event.

Proxy: provides placeholder for another object to control access, reduce cost and reduce complexity.
State: allows object to alter behaviour when internal state changes.


Strategy: allows one of family of algorithm to be selected on-the-fly at runtime.


Template method: defines skeleton of algorithm as abstract class allowing subclasses to provide concrete behaviour.


Visitor: separates algorithm from object structure by moving hierarchy of methods into one object.
