Case Study: A store requires software system for managing laptop rentals, customers can rent only one laptop at a time.

First Solution

First Review
  • Only single class that is specific to needs of laptop rental application.
  • Unlikely that Store class could be reused as part of developing another application.
  • Can’t add new functionality e.g. renting other hardware.
  • Policies could change e.g. customers can rent more than laptop at a time.


Second Solution

Second Review
  • Decomposing solution space into separate classes makes design more flexible.
  • Store depends on both Laptop and Customer so can’t be reused without them.
  • Laptop and Customer share mutual dependancy, one can’t be used without the other.



Third Solution

Third Review
  • Store class modified to break dependancies between Laptop and Customer - can be used in any sales application.
  • Classes have little logic.



Fourth Solution


Fifth Solution

Flexibility: Consider...
  1. Adding more of same functionality - allowing laptop rental application to handle rentals of other hardware.
  2. Adding different functionality - allowing laptop rental application to support purchases as well as rentals.
  3. Changing existing functionality - policy change such as allowing customers to rent multiple hardwares or changing price.
