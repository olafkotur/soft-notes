Architectural styles
  • Mechanisms or templates for structuring system architectures.
  • Description of good design practice, which have been tried and tested.
  • Include information about what they are and when they’re not useful.
  • Represented using tabular and graphical descriptions.

Architecture affects system characteristics
  • Performance: Localise critical operations and minimise communications, user large rather than fine-grain components.
  • Security: Use layered architecture with critical assets in inner layers.
  • Safety: Localise safety-critical features in small number of subsystems.
  • Availability: Include redundant components and mechanisms for fault tolerance.
  • Maintainability: Use fine-grain, replaceable components.

Architectural conflicts
  • Using large-grain components improve performance but reduces maintainability.
  • Introducing redundant components improves availability but makes security difficult. 
  • Localising safety-related features adds communication thus degraded performance.
  • Small-grain components improve reuse but degrade performance.

Client-server style



Two-tier client server architectures: System implemented as single logical server plus an indefinite number of client that use that server.
  • Thin: Presentation layer is implemented on the client (DBMS / Processing on server). 
  • Fat: Some or all processing is carried out on client (DBMS on server).

Three-tier architectures
  • Each of application architecture layers may execute on separate processors. 
  • Allows for better performance than thin approach and simpler to manage than fat.
  • More scalable architecture - demands increase, extra servers can be added.
