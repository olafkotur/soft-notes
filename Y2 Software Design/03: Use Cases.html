Use Case: Behavioural model of a system, describes how system responds to stimulate outside of system, consists of case diagram and use case descriptions.
  • Describe scenarios or sample execution scenarios for the system under development.
  • Mainly textual so easy to use as way to communicate between stakeholders. 
  • Used to derive test cases.



Case Diagrams
Case Descriptions
Case diagrams serve as high level reference, consists of different components.
 Case descriptions are developed as scenarios, must include a flow of events, primary scenario and secondary scenario(s).

Include vs. Extend
  • Include: Used to extract use case fragments that are duplicated in multiple cases.
  • Extend: Used when use case conditionally adds steps to another use case.




Use Case Guidelines
  1. Avoid Excessive Structuring: Only use includes and extends when necessary.
  2. Avoid Functional Decomposition: Congregate items into one to avoid thick density.
  3. Use Cases in Testing: Alistair Cockburn notes that UCs can be written in levels:
    1. Cloud level - Higher level summary, typically a business goal.
    2. Sea level - User goal, key characteristics, single sitting, user gets result.
    3. Fish level - Usually the include or extend use cases.
    4. Bottom-of-Ocean level - too low.
  4. Non-Functional Requirements: Don’t appear in use cases, reason why detractors criticise use cases, can be added using annotations e.g. support 300 million users.
