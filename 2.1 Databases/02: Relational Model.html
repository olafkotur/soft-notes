Relations
  • May be thought of as table of values, each row in table represents collection of related data values.
  • Table name and column names used to help interpret meaning of values in each row.



Domain: Set of atomic values (each value in domain is indivisible).
  • Essentially the column that a table has.
  • Domains also have data types i.e. Integers, Strings...



Notations
  • Relation schema (describes relation) R of degree n is denoted by R(A1, A2, …, An)
  • N-tuple t in relation is denoted by t = < V1, V2, …, Vn > where Vi is value corresponding to attribute A.


Degree of Relation: Number of attributes n of its relation schema
  • Example a degree of 4 would mean that a student may have the following attributes:
  • Name, Age, RegNum, Major


Bit about Tuples
  • Canny have duplicates
  • The order doesn’t matter as multiplication in mathematics does not care about ordering.
  • However, ordering values of tuples (in columns) is significant as the position under correct column is obviously important.
  • Able to have a domain and a value in a tuple e.g. ( Major, Software Eng. )




Query Template: All the values that we know and don’t know - helps to determine what needs to be requested.

Tuple Match: All fields must match the query template in order for it to be a tuple match. 
  • Don’t have to have all values as those we don’t know can be treated as joker cards.


Superkey: Any sets of attributes that will define something that is unique. All relations have at least 1 superkey.
  • Can have redundant attributes - doesn’t contribute to property of unique identification.
Keys
Candidate Key
Unique combination of attributes that can be used to uniquely identify record without referring to any other data. Each table may have more than one candidate, only one is chosen are primary key.
Primary Key
Unique identifier that can be used to identify a record without referring to any other data.
Foreign Key
Uniquely identifies a primary key in another table. Connects more than one tables together.
