Case study: Employee record system.




Relational File
  • Number of records, one after the other based on set of bytes.
  • Each record has defined set of related attributes.



Adding new data
  • All of the code written for this DB knows the existing format.
  • If you add a new field then format changes thus making all the other code incorrect.
  • Example, adding marital status now makes format 16 bytes.



Solution - Create new file
  1. However, two files now need to be maintained.
  2. Performance penalty accessing multiple files.
  3. Old code has no knowledge of new fils.
  4. Danger of data duplication.
  5. Adding or deleting records is more of an issue - if you delete one record e.g. employee then all the corresponding records in other files must be updated.


Hence, Database: We can fix all these issues by abstracting:
  1. Data Structure
  2. Data values
  3. Data access




Data Independence
  • Using DBMS we can the logical view as presented to applications and the physical layout on backing store as manipulated by DBMS.
  • When adding new information to data set, can be done to physical layout without ill effects to existing applications.
  • This is done by logical level only able to access information by queries from physical level.


Data Abstraction
  • Conceptual representation of data, hides details of how data is stored.
  • Uses logical concepts such as objects, properties and inter-properties.
  • Offers application software a consistent interface including most efficient mechanisms to access the physical data.
  • Self-describing: Data definitions, storage structure of data items, and constraints stored in system catalogue.

Views: Accessing same records from separate applications



Multiple Concurrent Access
  • Concurrency control ensures several users trying to update same data do so in controlled fashion so results of updates are correct e.g. reserving a plane seat.
  • Systems that require concurrent transactions are known as on-line transaction processing (OLTP) systems.

Other features of DBMS
  1. Restricting unauthorised access
  2. Multiple user interfaces e.g. query languages, APIs, GUIs, WWW
  3. Representing complex relationships among data
  4. Enforcing integrity constraints
  5. Backup and recovery

Definitions
Entity
Specific object, real or abstract, about which we need to store attributes and relationships e.g. person, car, company.
Attributes
Properties describing an entity, e.g. student will have age, name, major, minor...
Relationships
Associations among entities e.g. Person X married to person Y.
Record
Most common data aggregation mechanism, collection of related data values or items. 
Schema
Roadmap for data eventually stored

Limitations of DBMS
  • High initial investment in hardware, software and training.
  • Need to provide security, concurrency control, recovery and integrity.

When not to use DBMS
  1. Database and applications are simple, well-defined and not expected to change.
  2. Stringent real-time requirements for some programs that may not be met because of DBMS overheads.
  3. Multiple user access to data not required.

Database
DBMS
Logically coherent collection of data with some inherent meaning.
Collection of programs facilitating definition of database, construction of database and manipulation of database.


Actors in Database system environment
Database Administrator
  • Authorises access to database
  • Coordinates and monitors its use
  • Acquires software and hardware resources as required
Database designers
  • Identify data to be stored in database
  • Choose appropriate structures and constraints to represent and store data.
Software Engineers
  • Identify end user requirements
  • Implement, test, debug, document, maintain
Casual end users
  • Occasionally access database but may need different info each time
Naive or parametric end users
  • Make up sizeable portion of database end users.
  • Use canned transactions e.g. bank tellers, airline reservations.
Sophisticated end users
  • Implement their own applications e.g. engineers.
Stand-alone end users
  • Maintain personal databases using off-shelf systems e.g. Microsoft Money
