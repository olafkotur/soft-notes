FUN: Digital Systems Introduction

Computer

  • Automatic electronic apparatus for making calculations or controlling operations that are expressible in numerical or logical terms, and ability to run arbitrary programs stored in memory.
  • An ‘adding machine’ would not be classed as a computer as it only carries out one specialised function, therefore is not programmable.
  • Doesn’t have to be electronic.

FUN: Computer Architecture

Architecture Overview

  • Science of construction of any kind for human use.
  • Computer architecture e.g. hardware
  • System architecture e.g. linking computers with network
  • Software architecture e.g. OS, WWW, MS Office
  • Most popular is Von Neumann Architecture → → →

Harvard Architecture

  • Alternative architecture to the Von Neumann.
  • This simply means that the instruction memory and data memory are separate components.

Central Processing Unit (CPU)

  • Control unit plays the role of the central executive allocating resources and tasks to its slave components.
  • The ALU & Control Unit often have cache memory, this speeds up the process of some tasks by changing the location of the tasks. Using cache memory technology for standard RAM would be expensive as it comes from rare resources.

Evolution of computer architecture

50s-70s

  • Diverse hardware (mostly Von Neumann)
  • Supercomputers / Mainframes / Minicomputers
  • Different system software - apps had to be re-written for each

80s

  • More companies used computers, e.g. Bill Gates, Microsoft

80s-90s

  • IBM Personal (Micro) Computer, 1981, open (non-proprietary) architecture, evolving parallel with Intel x86 microprocessor family. Gained market dominance, caused death of mini-computer market.  Led to era of PC adoption and range of OS.

00s-10s

  • Computers are seen everywhere (Internet of things), people use watches, phones, laptops.

Modern architecture

  • Modified to increase speed of processing.
  • Popular measures of speed are; clock rate e.g. 1.87GHz CPU makes 1.87 billion ticks/s (different type of instructions may take different number of ticks).
  • Millions of instructions per second (MIPS) the higher the better, but different instructions take different numbers of ticks.
  • Floating point operations per second (FLOPS) measure of computer speed e.g. fastest computer in 2013 reached 33.863 petaFLOPS.

Limiting factors on speed

  • Density limitations - Number of transistors per square inch. Moore’s Law (1965 and 1975) predicts number of transistors on silicon will double every two years.
  • Power limitations - ⅓ of CPU power used to propagate clock signal around CPU. Power and heat problems increase as clock rate increases.

Parallel architectures

  • Demand is increasing for faster computation power, we overcome this by adding multiple cores to a single CPU. This increases efficiency, but complicates system software.
  • Allows Coarser-grained parallelism: clustering - linking computer using high-speed networks (blade servers) e.g. Google or Facebook servers, often in cold places to be more heat efficient.

  • Any data on computer is just a representation that maps to small, positive, numbers e.g. a code for characters could be A=1, B=2, C=3...

FUN: Boolean Logic

Overview

  • Sentence that is either true or false i.e. Boolean 1 or 0.
  • We can represent this in truth tables which show all possible logic combinations.

Inverted logical operations

  • NAND & NOR commonly used instead of normal gates.
  • Reason: Universal, any binary logic circuit can be built entirely from NAND or NOR gates.
  • Using only NANDs (or NORs) is cost-effective as you only need one component.

Karnaugh maps

  • Grid in which each square represents one possible combination of inputs.
  • Columns and rows are ordered so that only one input ‘changes’ from col-to-col and from ‘row-to-row’
  • They can also wrap from ‘left-to-right’ and ‘top-to-bottom’

Using a KM

  1. Pick an appropriate template with enough inputs.
  2. Write down all occurrences.
  3. Keep only the ones that repeat in given groups.
  4. Ensure to have the smallest amount of groups possible.

FUN: Instruction Set Architecture (ISA)

Overview

  • View of processor that is seen by programs being executed by that processor)
  • Provides commands to the processor, telling it what to do. Making it essentially machine language, commands include:
  • ADD: Add two numbers together
  • COMPARE: Compare numbers
  • IN: Input information from device e.g. keyboard
  • JUMP: Jump to designated RAM address
  • JUMP IF: Conditional statement that jumps to designated RAM address.
  • LOAD: Load information from RAM to CPU.
  • OUT: Output information to device e.g. monitor
  • STORE: Store information to RAM

Accumulator-based ISAs

Most instructions operate on two operands - register and memory location.

General purpose registers ISAs

Multiple registers, most instructions can access registers and memory locations in various combinations.

Reduced Instruction Set Computers (RISC)

Small number of instructions with common operand format. All instructions apart from LOAD and STORE operate on registers. Typically, instructions take only 1 clock cycle to execute.

Stack-based ISAs

Instructions can only access operands at top of stack - PUSH and POP instructions used to manipulate stack.

Carry-select adders

  • Slow because each stage must wait for carry bit from previous stage.
  • We can split the problem by adding the lower bits and upper bits independently.
  1. Add lower bits as normal.
  2. For the upper bits use two sets of full adders, one that assumes the result of lower bit is 1, other assumes 0.
  3. One answer is available, select one or the other.

Status flags

Overflow flag

  • If the sign bits are the same, but the result is different sign, we have an arithmetic overflow error.
  • The flag allows us to see which operation has an overflow error and allocate an appropriate error correction technique.

FUN: Memory

Memory

  • Main purpose is to store bits, there are two types of volatile memory.
  • Dynamic: Variable size, used for main memory → bits stored by charging or discharging device. Read by sensing presence or absence of charge.
  • Static: Fixed size, used for registers and caches.

Byte-ordering

  • Big-endian: Location with lowest memory holds the most significant byte (MSbyte).
  • Little-endian: Location with lowest memory holds the least significant byte (LSbyte).

Combinational logic and sequential logic

  • Combinational logic outputs are purely a function of its inputs
  • Sequential logic outputs are purely a function of its inputs and outputs.
  1. Involves feedback of the outputs and inputs.
  2. The feedback is the hook on which we hang memory

S-R flip flop

  • Remember which of 2 possible states it is in.
  • High S pulse → Q = 1 state (set)
  • High R pulse → Q = 0 stage (reset)
  • Not allowed to have both high.
  • When S/R returns to low, Q remains, thus flip-flop remembers state.

Clocked D-type flip flop

  • SR flip flop has distinct SET and RESET inputs. It’s better to have a , single input that sets (1) and resets (0).
  • Cannot tell flip flop when it should store input data.
  • We can use a D-type as a fix as it can latch in a 1 and 0.
  • To latch 1 you need two high inputs
  • To latch 0 you need at least one low input

Master slave-flip flop

  • We need close control when data leaves and arrives flip flop as otherwise we will overwrite stored bits before reading them.
  • Made up of two D-type flip flops.

Buses

  • Bundles of wires, one for each bit.
  • Address: Carries address of memory allocation from CPU to memory.
  • Data: Transfers data between components and other computers.
  • Bus wires we shared so we must ensure only one active output at a given time.

FUN: Control Unit

Overview

  • ‘Little program’ running inside the processor endlessly which executes the fetch, decode, execute cycle.
  • Controlled by a clock, gives it regula, timed electrical pulses (ticks).
  • One instruction may require multiple ticks depending on complexity of tasks.

Controlling main memory

  • Instruction register (IR) holds instruction currently being executed.
  • Memory address (MAR) holds current instruction address.
  • Memory data register (MDR) holds current instruction data going to and coming from memory address in the MAR.

Fetch-decode-execute cycle

FSM vs microcoded CUs

  • Finite state machine (FSM) - hard wired sequential logic, built directly in terms of NANDS. High performance, but expensive and hard to evolve.
  • Microcode - sequence of micro-instructions in a micro-memory. Lower performance but very flexible in terms evolving the system.

Pipelining

  • Method to exploit inherent parallelism inside CU to speed up FDE cycle.
  • Split cycle into n-sub stages e.g. human workers on factory production line.
  • Leads to multiple disadvantages (hazards)
  1. Control hazards: Control-transfer instruction changes flow of execution.
  2. Data hazards: Instruction ‘n’ depends on result from instruction ‘n-1’.
  3. Structured hazards: Two parts of pipeline need access to same hardware.

FUN: Input and output system

I/O challenge: Speed-gap

  • Devices often mechanical, run orders of magnitude slower than CPU e.g. we can’t slow down CPU to communicate with keyboard.

I/O challenge: Device diversity

  • Data access modes e.g. read/write only random access.
  • Device specific operations e.g. resolution, time, focus.
  • Device drivers group ‘somewhat’ similar devices together e.g. all hard drives are essentially the same.

Two-fold classification (device types)

  1. Character devices: Send and receive one byte at a time e.g. keyboard.
  2. Block devices: Send and receive multi-byte block at a time e.g. hard drive.

Disk latency

  • Latency more important than raw data transfer speed.
  • Seek time → time for head to move input to require cylinder.
  • Rotational delay → time for cylinder to ‘come around’.

Traditional two-fold class of processor support

  1. Isolated I/O → dedicated pins for devices

e.g. Intel x86 Instructions port addresses IN / OUT.

  1. Memory mapped I / O → within CPU’s linear memory addresses.

Data transfer errors

  • Electrical noise (long cables), mechanical stresses.
  • Code for error detection and correction.
  • Correction: Always expect even no. bits (0s)  or expect odd no. bits (1s).

ASS: Exceptions, Interrupt and I/O

Coprocessors

  • Extends functionality of MIPS while sharing instruction fetch and execution control logic of CPU.
  • CP0 - System Coprocessor: Abstraction of functions necessary to support OS, includes exception handling, memory management, scheduling.

Exception

  • Any unexpected change in flow control e.g. dividing by zero.
  • Synchronous Exception: Caused by instruction in running program e.g. invalid memory addresses in load, store, trap instructions.
  • Asynchronous Exception: Caused by I/O device requesting the processor (hence known as Hardware Interrupt).

Exception handling

  • Control transferred to an exception handler, in MIPS it is handled in CP0.
  • Shouldn’t write too much code in exception handler as we want to get out of it asap.
  • Best example of simple handling method is to skip the offending instruction.

Interrupts vs Polling

  • Interrupt: When device has an input e.g. keyboard is pressed, an interrupt is sent to CP0 and then dealt with (may need some sort of input hierarchy).
  • Polling: Microcontroller checks whether there is an input available throughout all devices (can waste a lot of time clock ticks if there are no available inputs).

ASS: Scheduler

Process scheduling

  • Process: Code defining behaviour (instructions), make instance and use registers to execute program. State of system is what runs the machine.
  • Periodic timer: Interrupt running processes. Save the state, deal with the interrupt and then return to restored process.
  • Allows us to prioritise tasks e.g. make certain tasks run more often than others.


Visualisation

  • Red shows tasks that the processor carries out that are seen as useful, the grey areas are non-useful tasks.
  • This shows that processor can waste time processing non-useful tasks, can speed up with software by speeding up switching between interrupts.

Virtual memory

  • Virtual memory space mapped onto real memory where a process is not aware of details of physical memory (only sees a chunk of memory starting at address 0).
  • The OS, with the help of memory management unit (MMU) manages mapping of VM.

BUG: Assembling programs

Executable creation

  1. Preprocessing: Statements e.g. C code
  2. Compilation: Transform C into assembler source code, depends on machine architecture.
  3. Assembly: Creates machine code stored in object file i.e. binary code.
  4. Linking: Combines several object files together, this is done by doing ‘gcc hello.o’

Static linking

  • Program and library that it’s linked against are combined together by linker, usually have the extension .a
  • Binding between program and library is fixed i.e. need to link again to change library.
  • Advantage: You know which libraries are used when compiling and everything is present when you copy over programs.
  • Disadvantage: Programs take a lot of disk space given that libraries are often big.

Dynamic linking

  • Program and library that it references are not combined together by linker, usually have the extension .so
  • Linker places information into executable that tells loader which object module code is in and which runtime linker should be used to bind references.
  • Advantage: Enables small file sizes on disk, libraries can be updated without re-assembly and more than one program can share the same library.
  • Disadvantage: Libraries may change, may not always understand impact on program.

Process loading

  • In a Linux environment, a process is loaded from a file in ELF format.
  • OS loader loads data into main memory which performs memory and access validation and process setup.
  • Allocate memory.
  • Copy address space from secondary to main memory.
  • Copy program arguments to stack.
  • Initialise registers e.g. the stack pointer.
  • Jump to start routine i.e. jump to main().

BUG: Linux

Boot Process

Stage 1: BIOS

Program counter of CPU initialised by power-on event to BIOS. Performs basic initialisation of hardware. Initial code includes the boot loader code, kernel code of stepping stone OS, kernel code of target OS.

Stage 2: Bootloader

Load system kernel image and initial ram disk, temp root file system (initrd) to memory and hands control over to them (fetching the OS).

Stage 3: Debian

Run system kernel with root filesystem in memory. Executes ‘init’ script.

Stage 4: Debian

Root filesystem is switched from memory to the one on the disk. Init program is executed to perform main boot process of starting programs.

Miscellaneous

  • SysV-style init: Standard process for controlling which program ‘init’ script launches or halt when initialising a runlevel → located in the /etc/rc.d/ directory.
  • Login: Logins from terminals and console are provided by ‘getty’ program, reads username and runs login program that reads password.
  • Authentication: Provided by PAM (Pluggable Authentication Modules), uses 3 config files → ‘etc/passwd’ user account info, ‘etc/shadow’ secure user account info, ‘etc/group’ group information.
  • Filesystem: Files organised in directories in a tree rooted at ‘/’
  • Shell: Any program that users use to type commands e.g. bash, tcsh, dash, csh.
  • X Window System: UNIX and Linux don't incorporate user interface into the kernel.

BUG: C Preprocessor

Overview

  • Macro processor that is used automatically by C compiler to transform program before compilation. Defines macros which are shortcuts for longer constructs.

Initial processing

Input file is read into memory and broken into lines, different systems used to indicate end of line. Continued lines are merged, all comments replaced with single spaces.

Tokenization

File converted into preprocessing tokens, identifier is any sequence, number includes all integer/floats, string literals are string/characters.

Preprocessing language

Consists of directives to be executed and macros to be expanded e.g. you can in/exclude parts of program according to conditions.

Header files

Included in preprocessing using #include, use <file> to search for file, and “file” to search for your own file.

Object-like macros

Created using #define directive, e.g. #define BUFFER_SIZE 1024 is an abbreviation for token 1024. Written in uppercase by convention.

Function-like macros

Created using #define directive and foobar(), e.g. #define lang_init(). They can take arguments just like functions.

Stringification

Convert macro argument into string constant, parameters not replaced inside string constants but you can use ‘#’ instead.

Predefined macros

You can use these without definitions e.g. _FILE_ expands to name of current file in form of string constant, _LINE_ expands to current input line number.

Undefining and redefining macros

Created using #undef directive, takes single argument, the name of macro to undefine. Has no effect if the name is not a macro.
